<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Telegram -> Bot | Untrusted user messages from Telegram API |
| Bot -> Claude SDK | Prompts sent to Claude Code subprocess |
| Claude SDK -> Filesystem | Claude Code can read/write files in session_dir |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | _handle_message | accept | PTB validates bot token; user_id comes from Telegram API (trusted) |
| T-02-02 | Tampering | _envelope_message | mitigate | User text is wrapped in envelope metadata but NOT sanitized for injection — Claude Code's own safety filters handle prompt injection. Envelope format makes user text boundaries clear. |
| T-02-03 | Information Disclosure | _build_system_context | accept | Only injects user's own name/language/chat type — no secrets |
| T-02-04 | Denial of Service | _enqueue_or_run | mitigate | Per-user asyncio locks prevent message storms; one user cannot block others. Queue acknowledgment ("...Queued") prevents Telegram retry loops. |
| T-02-05 | Elevation of Privilege | build_options | mitigate | permission_mode="acceptEdits" limits Claude to file edits, not arbitrary code execution outside allowed_tools. Session dirs isolate per-chat state. |
| T-02-06 | Denial of Service | _stream_text | mitigate | Streaming throttle (0.5s/30char) prevents Telegram rate limit errors |
</threat_model>
