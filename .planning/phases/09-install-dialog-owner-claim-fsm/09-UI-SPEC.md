## Typography

All values from Voidnet ui-spec.md §2 and confirmed in `bot/dashboard/static/style.css`.

| Role | Size | Weight | Line Height | Color Token |
|------|------|--------|-------------|-------------|
| Body | 1rem (16px) | 400 | 1.5 | `--text-primary` |
| Label / small — code display | 0.875rem (14px) | 400 | 1.5 | `--text-secondary` — code display uses `ui-monospace, SFMono-Regular, Menlo, Consolas, monospace` |
| Micro / badge | 0.75rem (12px) | 400 | 1.4 | `--text-faint` or per-badge token |
| Heading (H2) — pairing code display | 1.3rem (21px) | 600 | 1.3 | `--text-secondary` for headings; `--text-primary` + `letter-spacing: .3em` for pairing code |
| Section label (H3) | 1rem (16px) | 400 | 1.3 | `--text-dim`, uppercase, `letter-spacing: .05em` |

Declared sizes: 12px / 14px / 16px / 21px (4 sizes). Declared weights: 400 (regular) and 600 (emphasis/heading/pairing-code) (2 weights). The pairing code display uses 21px at weight 600 with `letter-spacing: .3em` in `--text-primary` — visual weight is achieved via spacing, not a unique size or weight.

## Component Inventory

These are the specific UI elements Phase 9 adds or extends on the telegram-bridge config page (`/modules/telegram-bridge/config`). All use existing CSS classes from style.css — no new classes required.

### 1. Token Install Form (state: not installed)

Rendered by the config fragment when `has_token == false`.

```
.field
  label[for=token-input]  "Bot Token"
  input#token-input[type=password, placeholder="110201543:AAHdqTcvCH1vGWJxfSeofSz^MXFggAndnqQ"]
  .field-help  "Paste your bot token from @BotFather."
.btn.btn-success[hx-post="/api/modules/telegram-bridge/install"]  "Install Bridge"
```

- Input type is `password` (masks token on screen — SEC-01 defense in depth)
- `hx-post` targets `#status-toast` with `hx-swap="innerHTML"` for inline error/success feedback
- On valid token: backend redirects to config page showing running state (full page reload via `HX-Redirect` header)
- On invalid token: `.error` fragment replaces `#status-toast` (no page reload)

### 2. Status Strip (state: installed, running)

Existing `_fragments/status_strip.html` pattern. Shows:
- `.dot.dot-green` + "Running" text
- Bot username if available (e.g. `@my_animaya_bot`) in `--text-muted`

### 3. Pairing Code Block (state: claim_status == "pending")

New fragment `_fragments/pairing_code_pending.html`. Structure:

```
div#claim-status[hx-get="/api/modules/telegram-bridge/claim-status"
                 hx-trigger="every 5s"
                 hx-target="#claim-status"
                 hx-swap="outerHTML"]
  .panel
    .panel-header  "Claim Ownership"
    div[padding: 1rem 1.25rem]
      p  "Send this code to your bot in Telegram:"
      div.pairing-code  "{{ code }}"   ← 21px (1.3rem) w-600 letter-spacing:.3em --text-primary
      .progress[style="margin: .5rem 0"]
        .progress-bar[id=countdown-bar style="width: {{ pct }}%"]
      p.field-help  "Expires in {{ ttl_display }}  ·  {{ 5 - attempts }} attempts remaining"
      button.btn[hx-post="/api/modules/telegram-bridge/regenerate"
                 hx-target="#claim-status"
                 hx-swap="outerHTML"]  "Regenerate Code"
```

- Progress bar uses `--accent` fill, shrinks linearly as TTL drains
- When TTL < 2 minutes: progress bar switches to `--warn` fill (add inline `style="background: var(--warn)"`)
- `.pairing-code` is a new display-only element: `font-family: ui-monospace; font-size: 1.3rem; font-weight: 600; letter-spacing: .3em; padding: .75rem 1rem; background: var(--bg-base); border: 1px solid var(--border-subtle); border-radius: 6px; color: var(--text-primary); text-align: center;`

### 4. Claimed State Block (state: claim_status == "claimed")

New fragment `_fragments/pairing_code_claimed.html`. Replaces the polling div — polling stops automatically because this fragment has no `hx-trigger`.

```
div#claim-status
  .success  "Ownership claimed. You are the owner of this bot."
  button.btn.btn-danger[hx-post="/api/modules/telegram-bridge/revoke"
                        hx-confirm="Revoke ownership? The bot will be accessible to anyone until reclaimed."
                        hx-target="#claim-status"
                        hx-swap="outerHTML"]  "Revoke Ownership"
```

### 5. Unclaimed / No Code State (state: claim_status == "unclaimed", installed)

```
div#claim-status
  p.field-help  "No owner claimed. Generate a pairing code and send it to your bot in Telegram."
  button.btn.btn-success[hx-post="/api/modules/telegram-bridge/generate-code"
                         hx-target="#claim-status"
                         hx-swap="outerHTML"]  "Generate Pairing Code"
```

### 6. Error Fragment (inline, replaces #status-toast)

Reuses existing `.error` class:

```html
<div class="error">{{ message }}</div>
```

## Focal Points by FSM State

| FSM State | Primary Visual Anchor |
|-----------|----------------------|
| not-installed | Token password input field — cursor lands here on page load; Install Bridge button is the sole CTA |
| installed / unclaimed | "Generate Pairing Code" button — only interactive element in the claim block |
| pending | Pairing code display block (`div.pairing-code`) — large monospace digits are the dominant element; countdown bar provides secondary attention signal |
| claimed | Success confirmation text — static read state; Revoke Ownership button is visually de-emphasized (destructive, `.btn-danger`) to avoid accidental tap |
| error (any) | `.error` alert box — `role="alert"` ensures immediate focus; positioned directly below the triggering form element |

## Interaction States

### Token Install Flow

| State | Trigger | UI Change |
|-------|---------|-----------|
| Idle | Page load, not installed | Token form visible, Install button enabled |
| Submitting | User clicks Install | Button disabled, `.progress-bar.progress-indeterminate` shows above form |
| Error — invalid token | Backend returns error | `.error` in `#status-toast`, form remains editable, button re-enabled |
| Error — network | Backend returns network error | `.error` in `#status-toast`, same as above |
| Success | getMe valid, install complete | Full page reload via `HX-Redirect` to config page, now shows running state + unclaimed block |

### Pairing Code Flow

| State | Trigger | UI Change |
|-------|---------|-----------|
| Unclaimed | Install complete or revoke | "Generate Pairing Code" button visible |
| Pending | Generate clicked | Polling div replaces button block; 6-digit code displayed; progress bar starts at 100% |
| Claimed | Valid code received via Telegram | Poll returns claimed fragment; polling div replaced; success message + Revoke button |
| Expired | TTL exceeded (detected by poll endpoint) | Poll returns unclaimed fragment; user sees "Regenerate Code" button |
| Max attempts | 5 failures (detected by poll or bridge) | Poll returns unclaimed fragment; user sees "Regenerate Code" button |
| Regenerate clicked | User action | New code + new TTL; same pending fragment swapped in |
| Revoke clicked | User action | `hx-confirm` dialog; on confirm: POST /revoke; returns unclaimed fragment |

### HTMX Polling Contract

- Poll interval: every 5 seconds (`hx-trigger="every 5s"`)
- Poll endpoint: `GET /api/modules/telegram-bridge/claim-status`
- Stop condition: endpoint returns fragment without `hx-trigger` attribute (claimed or unclaimed)
- Polling div ID: `#claim-status` (stable across all states)
- Swap: `outerHTML` (entire div replaced, including the trigger attribute)

## Accessibility Contract

From Voidnet ui-spec.md §7 — all apply:

- All inputs have `<label for="...">` matching `<input id="...">`
- All buttons are `<button>` elements, not `<div>` or `<span>`
- Focus-visible ring: `outline: 2px solid var(--accent); outline-offset: 2px` (already in style.css)
- The pairing code display element is a `<div aria-label="Pairing code: {{ code }}" aria-live="polite">` so screen readers announce the code on generation
- The countdown help text uses `aria-live="polite"` so TTL updates are announced
- Revoke confirmation uses native `hx-confirm` (browser dialog) — accessible by default
- `.error` and `.success` boxes use `role="alert"` so errors are announced immediately
