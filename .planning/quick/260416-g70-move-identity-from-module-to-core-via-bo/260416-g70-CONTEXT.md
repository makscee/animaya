# Quick Task 260416-g70: move identity from module to core via BOOTSTRAP.md lifecycle - Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Task Boundary

Move the identity concern out of the modules system and into Animaya's core.
Replace the hardcoded `ConversationHandler` Q1/Q2/Q3 onboarding with a
BOOTSTRAP.md-driven natural flow: Claude reads instructions from a file
shipped with the repo, elicits identity details conversationally (matching
the user's first spoken language), writes `USER.md` / `SOUL.md` to the hub
when ready, then deletes the BOOTSTRAP.md to mark onboarding complete.

Reset for repeat testing: `git checkout BOOTSTRAP.md` + delete generated
identity files.

</domain>

<decisions>
## Implementation Decisions

### Identity becomes core, not a module
- DELETE `modules/identity/` directory entirely (manifest, install.sh,
  uninstall.sh, prompt.md, README.md).
- DELETE `bot/modules_runtime/identity.py` (ConversationHandler + sentinel
  helpers); or strip to just the USER.md/SOUL.md file-path constants if the
  assembler still needs them.
- REMOVE `build_onboarding_handler` registration from `bot/bridge/telegram.py`.
- Identity file paths (`~/hub/knowledge/identity/USER.md`, `SOUL.md`) stay
  the same — Claude writes them via the Claude Code SDK Write tool.

### BOOTSTRAP.md lifecycle
- **Location:** repo root, path `BOOTSTRAP.md`. Tracked in git.
- **Content:** instructions to Claude about how to initialize identity:
  - Mirror the user's first-message language (respond in Russian if they wrote Russian, etc.)
  - Get to know the user through normal conversation — no rigid Q1/Q2/Q3
  - Guidance on what to capture (name/handle, role/context, how they want
    to be addressed, what the assistant's persona/tone should be)
  - Trust Claude's judgement on when enough has been captured to commit
  - When ready: Write `~/hub/knowledge/identity/USER.md` and `SOUL.md`,
    then delete the working-tree copy of BOOTSTRAP.md to clear the flag
- **Trigger:** BOOTSTRAP.md's existence in the repo working tree IS the
  trigger. No `.pending-onboarding` sentinel, no placeholder markers needed.
- **System-prompt injection:** when BOOTSTRAP.md exists at repo root, its
  contents get appended to the Claude Code system prompt (via `claude_query.py`
  `build_options` or the assembler). When deleted, injection stops.

### Reset flow
- `scripts/reset-dev.sh` (already on LXC) must be updated to also run
  `git checkout BOOTSTRAP.md` inside the repo and remove
  `~/hub/knowledge/identity/` alongside the existing wipes.

### /identity command fate
- REMOVE. No Telegram-level re-onboarding command. The operator resets by
  running `reset-dev.sh` (or `git checkout BOOTSTRAP.md`) if they want to
  re-onboard.

### Tests
- DELETE tests that exercise `onboarding_start`, `_onboarding_q1/q2/q3`,
  `sentinel_filter`, `build_onboarding_handler`.
- DELETE or adapt tests that cover `modules/identity/` install/uninstall.
- ADD a minimal test verifying BOOTSTRAP.md content is injected into the
  system prompt only when the file exists.

### Claude's Discretion
- Exact wording of BOOTSTRAP.md (tone, length, how explicitly to teach
  Claude the language-mirror rule). Recommendation: short (50-150 lines),
  voiced as operator instructions, not as a script the user sees.
- Whether to keep the identity file paths injected into the system prompt
  unconditionally (as the module did) OR only when non-empty. Recommendation:
  keep unconditionally — already-onboarded bots benefit from identity
  context, and reading empty files is fine.
- Whether the assembler needs any module-system changes to cope with
  identity's removal (registry entries, CLAUDE.md assembly template).
- Migration for bots already in "pending" state: acceptable to require a
  manual reset (`reset-dev.sh`). No auto-migration code.

</decisions>

<specifics>
## Specific Ideas

**BOOTSTRAP.md content sketch** (for the planner — not final wording):

```markdown
# Bootstrap

Until this file is deleted, you are onboarding a new operator for
Animaya. Follow this flow:

1. **Mirror their language.** If their first message is Russian, reply
   in Russian. English in English. Same for any other language.
2. **Get to know them through normal conversation.** Ask about who they
   are, what they do, what matters to them — but naturally, not as a
   checklist. React to what they say.
3. **Also shape the assistant's persona.** Ask what kind of assistant
   they want: tone, priorities, how they'd like to be addressed.
4. **When you have enough to work with**, write
   `~/hub/knowledge/identity/USER.md` (who they are) and
   `~/hub/knowledge/identity/SOUL.md` (your persona toward them). Then
   delete this BOOTSTRAP.md file to mark onboarding complete.

Trust your own judgement on "enough". Err toward capturing less but
correct, over more but invented.
```

**Repo layout after refactor:**
- `BOOTSTRAP.md` (new, at repo root)

- `bot/modules_runtime/identity.py` deleted (OR minimal shim if assembler still imports it)
- `scripts/reset-dev.sh` updated to restore BOOTSTRAP.md + wipe `~/hub/knowledge/identity/`
