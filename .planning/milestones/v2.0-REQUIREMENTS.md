# Requirements: Animaya v2.0 — Onboarding Polish & Bridge-as-Module

**Milestone:** v2.0
**Started:** 2026-04-15
**Status:** Roadmapped

## v2.0 Requirements

### Bridge-as-Module (BRDG)

- [ ] **BRDG-01**: Telegram bridge lives as `bot/modules/telegram-bridge/` module; `bot/main.py` starts dashboard only and defers polling to module lifecycle
- [ ] **BRDG-02**: Install dialog captures bot token and validates via Telegram `getMe` before persisting to module config
- [ ] **BRDG-03**: Master disable toggle stops polling without uninstalling the module (re-enable resumes polling)
- [ ] **BRDG-04**: Uninstall stops polling, purges module `state.json`, revokes owner, removes all module artifacts

### Owner-Claim Flow (CLAIM)

- [ ] **CLAIM-01**: Install emits a 6-digit pairing code with 10-minute TTL, capped at 5 attempts, verified with `hmac.compare_digest`
- [ ] **CLAIM-02**: Dashboard can regenerate the pairing code on demand, invalidating any prior code
- [ ] **CLAIM-03**: Owner can revoke ownership from dashboard; module returns to pending-claim state
- [ ] **CLAIM-04**: Owner `user_id` persisted only in module `state.json`; `TELEGRAM_OWNER_ID` env gate removed

### Non-Owner Access (ACC)

- [ ] **ACC-01**: Non-owner policy is a tri-state enum (`drop` / `flag` / `open`) with `drop` as default
- [ ] **ACC-02**: Flagged senders have their messages prefixed with `[non-owner user NAME (id=N)]:` in the Claude prompt
- [ ] **ACC-03**: Non-owner turns run with a restricted `allowed_tools` allowlist (no Edit/Write/Bash by default)

### Tool-Use Display (TUI)

- [ ] **TUI-01**: Per-bridge tool-use display setting with three modes: `temporary` / `persistent` / `hidden`
- [ ] **TUI-02**: `temporary` mode deletes tool-use messages from Telegram when answer streaming completes
- [ ] **TUI-03**: `temporary` is the default on fresh install

### Identity Pre-Install (IDN)

- [ ] **IDN-01**: Identity module pre-installed on first boot via idempotent `first_boot_install` hook (uses `open(path, "x")` so existing Hub data is never clobbered)
- [ ] **IDN-02**: Dashboard identity config page lists identity files with raw-markdown textarea editors
- [ ] **IDN-03**: Identity edits save to Hub and are reflected in the bot's system prompt without restart

### Dashboard Chat + Hub Files (DASH)

- [ ] **DASH-01**: Unified dashboard page pairs a chat panel with a collapsible Hub file tree (single page, not two)
- [ ] **DASH-02**: Chat streams responses via SSE and renders tool use inline (matching bridge tool-display style)
- [ ] **DASH-03**: File tree walks `~/hub/` (knowledge + data), is collapsible by directory, read-only in v2.0, dotfiles hidden by default
- [ ] **DASH-04**: File tree rejects any path outside `~/hub/` via `Path.resolve().is_relative_to(HUB_ROOT.resolve())` + symlink rejection; a DENY set blocks `.git/hooks/`, `.ssh/`, `.env*`

### Security & Isolation (SEC)

- [ ] **SEC-01**: Bot token never leaves the server — redacted in `/api/modules` GET responses and in logs (use pydantic `SecretStr`)
- [ ] **SEC-02**: Claude session keys are namespaced per UI source (`tg:<user_id>` vs `web:<user_id>`) to prevent cross-UI memory drift

## Future Requirements (Deferred)

- File tree edit mode (v2.0 is read-only)
- QR-code pairing alternative to 6-digit
- Multi-owner / team-bot support
- Chat history surfaced in Telegram (cross-UI conversation viewer)
- Module marketplace / third-party module install
- Identity rich-preview (rendered markdown + system-prompt diff)

## Out of Scope

- BotFather automated bot creation — violates Telegram ToS
- OAuth for Telegram — Telegram has no OAuth
- Per-user allowlist UI — `drop`/`flag`/`open` enum is the entire non-owner surface for v2.0
- Non-owner sandbox subdirectory — deferred; restricted `allowed_tools` is sufficient for v2.0
